프로젝트 개요 - 시작 배경

실무 니즈에서 출발

  • 국토교통부 통계누리 사이트의 "통계이야기" 페이지

  • 어떤 통계를 선택할지 방향 잡기 어려움
  • 통계표만 봐서는 기획 포인트 파악 힘듦

"AI 인사이트만 있어도 기획할 때 방향 잡기가 편하지 않을까?"

해결 방안

  • 최신 통계 1달치만 자동 수집
  • AI가 각 통계의 핵심 인사이트 자동 생성

향후 목표

  • HTML 차트 기반 시각화 구현
  • 카드뉴스 형태 분석 결과 제공
  • 실제 업무에 바로 활용 가능한 수준으로 발전

주요 기능 - AI 채팅 (RAG) NEW

RAG 기반 대화형 분석

위치: 분석 결과 페이지 내 아코디언 UI

작동 방식

  1. 사용자가 통계 데이터에 대해 질문
  2. ChromaDB에서 질문과 관련된 데이터만 검색 (상위 8개)
  3. AI 인사이트 + 검색된 데이터 → Ollama에게 전달
  4. 컨텍스트 기반 정확한 답변 생성

주요 특징

  • RAG (Retrieval-Augmented Generation) 방식
  • 대화 히스토리 유지
  • 실시간 응답 (로컬 LLM)
  • 데이터 외부 유출 없음

핵심 기술 - AI 분석 엔진 (Ollama)

Ollama 로컬 LLM 활용

오픈소스 로컬 AI로 데이터 프라이버시 보장

기술 스펙

  • 모델: Llama 3.1 (8B Instruct)
  • 실행 환경: 로컬 서버 (CPU 전용)
  • 장점: 비용 0원, 데이터 외부 유출 없음
  • 제약사항: 16K 토큰 제한, CPU 성능 제한

극복 방법

  • 통계표별 분할 분석으로 토큰 제한 회피
  • CPU 최적화 모델 사용

분석 프로세스

  1. 데이터 구조 분석 - 컬럼, 연도, 단위 파악
  2. 기본 통계량 계산 - 평균, 중앙값, 최대/최소값
  3. 시계열 트렌드 분석 - 증가/감소 패턴 파악
  4. 주요 변화 시점 탐지 - 급격한 변화 지점 식별
  5. 통계표별 개별 인사이트
  6. 종합 인사이트

성능 한계

  • 통계표당 3~10분 소요 (CPU 환경)
  • Claude/GPT 대비 30-40% 품질 차이
  • 하지만 업무 목적으로는 충분

개발 방법론 - 바이브 코딩

개발 규모

2-3주

개발 기간

40-60h

작업 시간

1명

개발 인원

20+

개발 일지

바이브 코딩이란?

AI와 대화하며 코드를 작성하는 방식

  • Claude (Cursor AI)와 실시간 대화
  • 문제 설명 → AI가 코드 제안 → 테스트 → 개선
  • 전통적인 검색 & 복붙이 아닌 협업 개발

장점

  • 비개발자도 복잡한 시스템 구축 가능
  • 디자이너가 풀스택 개발 완성
  • 전통적 방식 대비 2-3배 빠름

한계점

  • AI가 제안한 코드를 이해 없이 사용하면 위험
  • 복잡한 버그는 직접 디버깅 필요
  • 프로젝트 전체 구조 파악은 개발자 몫
  • 주석과 문서 부족 - 향후 개선 필요

개발 과정 - 주요 챌린지

1. IBSheet 동적 테이블 크롤링

문제점

  • 특수 JavaScript 라이브러리로 렌더링
  • DOM에 직접 접근 불가

해결책

  • Selenium으로 완전 로딩 대기
  • JavaScript 실행 완료 후 DOM 파싱

2. 대용량 데이터 처리

문제점

  • 수십 년치 통계 데이터
  • 메모리 부족 위험

해결책

  • 캐시 시스템 구축
  • 점진적 로딩 (Lazy Loading)
  • Pandas 최적화

3. AI 분석 - 하드웨어 제약 극복

하드웨어 환경: CPU만 있는 노트북 (GPU 없음)

문제점

  • 처음 시도: 통계 전체 데이터를 한 번에 분석
  • 결과: 토큰 양 초과로 실패 (16K 제한)

해결책 (핵심!)

  • Ollama 로컬 LLM 활용 (비용 0원)
  • 통계표별 분할 분석 - 통계 전체 → 개별 통계표로 분할, 각 표별 인사이트 생성 후 종합
  • 프롬프트 최적화

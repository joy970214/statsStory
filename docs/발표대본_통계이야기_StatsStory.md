# 📊 통계이야기 (StatsStory) 발표 대본

## 표지 슬라이드

안녕하세요. 오늘 소개드릴 프로젝트는 "통계이야기"라는 AI 에이아이 기반 국토교통부 통계 분석 시스템입니다. 

---

## 📑 목차

발표는 크게 16개 파트로 구성되어 있습니다. 

먼저 프로젝트 개요와 시스템 아키텍처(Architecture)를 말씀드리고, 사용한 기술 스택(Tech Stack)과 워크플로우(Workflow)를 설명하겠습니다. 

그 다음 주요 기능과 핵심 기술들을 소개하고, 개발 방법론인 바이브 코딩(Vibe Coding)에 대해 말씀드리겠습니다. 

마지막으로 프로젝트 성과와 한계점, 그리고 배운 점을 공유한 후 질의응답 시간을 갖도록 하겠습니다.

## 1. 프로젝트 개요

### 프로젝트 시작 배경

이 프로젝트는 제 실무 니즈에서 출발했습니다. 

저는 국토교통부 통계누리 사이트의 "통계이야기"라는 페이지에 올라가는 카드뉴스를 매달 제작하는 업무를 담당하고 있습니다. 예를 들어 8월 통계이야기를 만들려면, 7월에 올라온 통계 중에서 선택해야 합니다.

그런데 몇 가지 문제가 있었습니다. 첫째, 저는 디자이너 배경이라 통계 분석 경험이 부족했고요. 둘째, 한 달치 모든 통계를 일일이 확인하기가 너무 불편했습니다. 셋째, 어떤 통계를 선택할지 방향을 잡기가 어려웠고, 통계표만 봐서는 기획 포인트를 파악하기가 힘들었습니다.

### 해결 아이디어

그래서 이런 생각을 했습니다. "AI 에이아이 인사이트(Insight)만 있어도 기획할 때 방향 잡기가 편하지 않을까?"

최신 통계 한 달치만 자동으로 수집하고, AI 에이아이가 각 통계의 핵심 인사이트를 자동 생성해주면, 통계별 특징을 한눈에 비교해서 어떤 통계로 카드뉴스를 만들지 빠르게 결정할 수 있을 것 같았습니다.

다만 현실적인 제약도 있습니다. 통계당 인사이트 생성에 5분에서 10분 정도 소요되고, CPU 씨피유 환경의 한계로 실시간 분석은 어렵습니다.

### 향후 목표

향후 목표는 HTML 에이치티엠엘 차트 기반 시각화를 구현하고, 카드뉴스 형태의 분석 결과를 제공해서, 실제 업무에 바로 활용 가능한 수준으로 발전시키는 것입니다.

## 2. 시스템 아키텍처

시스템 아키텍처를 보시면, 크게 프론트엔드(Frontend)와 백엔드(Backend)로 구성되어 있습니다.

프론트엔드는 리액트(React), 타입스크립트(TypeScript), 테일윈드 씨에스에스(Tailwind CSS)로 개발했고, 사용자가 통계 목록을 보고, 실시간 진행률을 확인하며, 분석 결과를 시각화할 수 있습니다. 여기에 새로 추가된 AI 에이아이 채팅 인터페이스도 있습니다.

백엔드는 파이썬(Python) 패스트API(FastAPI)로 구축했고, 통계 크롤링, 데이터 가공, AI 에이아이 분석 처리, 그리고 RAG 래그, 리트리벌 어그먼티드 제너레이션(Retrieval-Augmented Generation) 기반 질의응답을 담당합니다.

하위에는 5개의 주요 서비스가 있습니다. 크롤러 서비스(Crawler Service)는 셀레니움(Selenium)으로 웹 크롤링을 하고, 올라마 서비스(Ollama Service)는 라마(Llama) 3.1 모델로 AI 에이아이 분석을 수행합니다. 캐시 스토리지(Cache Storage)는 JSON 제이슨 형식으로 수집 결과를 캐싱하고, 데이터 스토리지(Data Storage)는 통계 데이터를 저장합니다. 

그리고 새롭게 추가된 벡터DB 벡터디비, 크로마DB 크로마디비(VectorDB, ChromaDB)는 통계 데이터를 벡터화해서 저장하고, RAG 래그 기반 의미 검색을 지원합니다.

참고로 MCP 엠씨피 서버(MCP Server)는 코드에만 존재하고 실제로는 사용하지 않습니다. 향후 확장용으로 남겨둔 것입니다.

## 3. 기술 스택

### 프론트엔드 기술

프론트엔드 기술로는 리액트(React) 19 최신 버전을 사용했고, 타입스크립트(TypeScript)로 타입 안정성을 보장했습니다. 테일윈드 씨에스에스(Tailwind CSS)로 모던한 스타일링을 했고, 액시오스(Axios)로 HTTP 에이치티티피 통신을, 프레이머 모션(Framer Motion)으로 부드러운 애니메이션을 구현했습니다. 아이콘은 히어로아이콘즈(Heroicons)와 루시드 리액트(Lucide React)를 사용했습니다.

### 백엔드 기술

백엔드는 패스트API(FastAPI)라는 고성능 파이썬 웹 프레임워크를 사용했고, 셀레니움(Selenium)으로 동적 웹 크롤링을, 뷰티풀수프(BeautifulSoup) 4로 HTML 에이치티엠엘 파싱을 했습니다. 판다스(Pandas)로 데이터 처리 및 분석을, 파이댄틱(Pydantic)으로 데이터 검증 및 직렬화를 수행합니다.

AI 에이아이 분석 엔진으로는 올라마(Ollama) 라마(Llama) 3.1을 사용하고 있고, 새롭게 크로마DB 크로마디비(ChromaDB)라는 벡터 데이터베이스를 추가해서 RAG 래그를 지원합니다.

### 배포 및 확장성

현재는 로컬 환경에서만 실행되는 개인 프로젝트 수준이지만, 향후에는 도커 컴포즈(Docker Compose)로 원클릭 설치가 가능하도록 만들 계획입니다. 타 팀에서 사용 가능하려면 약 6개월 정도 소요될 것으로 예상됩니다.

폐쇄망 환경에서도 사용 가능한데, 올라마(Ollama)가 로컬 엘엘엠(LLM)이라서 모델만 사전에 다운로드하면 됩니다. 타 기관 통계로 확장하려면 사이트별로 크롤러를 개발해야 하는데, 사이트당 1주에서 2주 정도 걸릴 것 같습니다.

## 4. 핵심 워크플로우

핵심 워크플로우는 4단계로 구성됩니다.

### 1단계: 최신 통계 수집
먼저 국토교통부 통계누리에서 최근 한 달간의 통계를 자동으로 크롤링합니다. 매달 "통계이야기" 제작을 위한 후보 통계를 수집하는 것이죠.

### 2단계: 사용자 통계 선택
사용자가 관심 있는 통계를 클릭해서 선택합니다.

### 3단계: 실시간 데이터 수집 및 분석
선택한 통계의 메타데이터를 수집하고, 통계 개요나 용어정의 같은 정보를 가져옵니다. 그 다음 엑셀 파일을 자동 다운로드하는 것을 우선으로 하고, 안 되면 테이블 데이터를 직접 수집합니다. 마지막으로 로컬 AI 에이아이 기반으로 자동 분석을 수행합니다.

### 4단계: 결과 확인 및 활용
AI 에이아이 인사이트로 통계별 특징을 파악하고, 새로 추가된 AI 에이아이 채팅 기능으로 통계 데이터에 대해 질문할 수도 있습니다. 이를 바탕으로 어떤 통계로 카드뉴스를 만들지 빠르게 결정하고, 구조화된 데이터는 추후에 활용할 수 있습니다.

## 5. 주요 기능 - 통계 목록

주요 기능 첫 번째는 통계 목록 화면입니다.

최근 한 달간의 통계가 자동으로 표시되고, 국토계획, 교통, 주택 등 카테고리별로 필터링할 수 있습니다. 이미 수집이 완료된 통계는 체크마크로 표시되고, 실시간 서버 상태도 확인할 수 있습니다. 반응형 카드 레이아웃으로 모바일에서도 잘 보입니다.

## 5. 주요 기능 - AI 채팅 (RAG)

두 번째 주요 기능은 AI 에이아이 채팅입니다. RAG 래그, 리트리벌 어그먼티드 제너레이션(Retrieval-Augmented Generation) 기반 대화형 분석 기능입니다.

### 위치 및 작동 방식

이 기능은 분석 결과 페이지 내에 아코디언(Accordion) 형태의 UI 유아이로 제공됩니다.

작동 방식은 이렇습니다. 첫째, 사용자가 통계 데이터에 대해 질문을 입력하면, 둘째, 크로마DB 크로마디비(ChromaDB)에서 질문과 관련된 데이터만 상위 8개를 검색합니다. 셋째, AI 에이아이 인사이트와 검색된 데이터를 올라마(Ollama)에게 전달하고, 넷째, 컨텍스트(Context) 기반으로 정확한 답변을 생성합니다.

### 주요 특징

RAG 래그 방식의 가장 큰 장점은, 전체 데이터가 아니라 관련 데이터만 검색해서 사용하기 때문에 토큰 제한을 회피할 수 있고 정확도도 향상된다는 것입니다.

대화 히스토리도 유지되어서 이전 대화 맥락을 기억하고 연속된 질문이 가능합니다.

로컬 엘엘엠(LLM)으로 빠른 응답을 받을 수 있고, 데이터가 외부로 유출되지 않아 안전합니다.

### 사용 예시

예를 들어 "2023년 서울 주택 미분양 현황은?"이라고 물어볼 수도 있고, "전년 대비 증가율이 가장 높은 지역은?" 같은 구체적인 질문도 가능합니다. "최근 3년간 추이를 요약해줘"처럼 전체적인 흐름을 파악하는 질문도 할 수 있습니다.

## 5. 주요 기능 - 실시간 진행률 표시

세 번째 주요 기능은 실시간 진행률 표시입니다.

### SSE 기술

SSE 에스에스이, 서버 센트 이벤츠(Server-Sent Events) 기술을 사용해서 서버에서 클라이언트로 실시간 데이터를 푸시합니다.

### 표시되는 정보

현재 진행 단계가 무엇인지, 예를 들어 메타데이터 수집 중인지, 테이블 수집 중인지 표시됩니다. 진행률도 퍼센트로 표시되고 프로그레스 바(Progress Bar)로 시각화됩니다. 현재 작업 내용도 상세 메시지로 보여주고, 예상 소요 시간도 안내합니다. 언제든지 취소 버튼으로 작업을 중단할 수도 있습니다.

### 장점

이렇게 하면 작업이 멈춘 것처럼 보이지 않아서 사용자 경험이 좋고, 투명한 프로세스를 제공해서 사용자 이탈을 방지할 수 있습니다.

## 5. 주요 기능 - 분석 결과

네 번째 주요 기능은 분석 결과 화면입니다.

### 제공되는 분석 결과

세 가지 종류의 분석 결과를 제공합니다.

첫째, 기본 통계로 평균, 중앙값, 최댓값, 최솟값과 전체 데이터 포인트 수를 보여줍니다.

둘째, 구조화된 데이터로 연도별 원본 데이터를 JSON 제이슨 형태로 저장하고, 테이블별로 데이터를 분리해서 저장합니다. 캐시를 통해 빠르게 재사용할 수 있습니다.

셋째, AI 에이아이 인사이트입니다. 텍스트 기반으로 데이터 트렌드를 분석하고, 주요 변화 시점을 탐지하며, 통계표별 특징을 분석하고 통계적 의미를 해석해 줍니다.

## 6. 핵심 기술 - 동적 웹 크롤링

핵심 기술 첫 번째는 동적 웹 크롤링입니다.

### 도전 과제

국토교통부 통계 사이트는 복잡한 자바스크립트(JavaScript) 기반으로 되어 있어서, 동적 렌더링으로 일반 크롤링이 불가능했습니다. 테이블 구조도 복잡해서 병합 셀이나 다층 헤더가 있고, 데이터 형식도 IBSheet 아이비시트나 엑셀 등 다양했습니다.

### 해결 방법

이중 전략을 사용했습니다.

우선적으로는 엑셀 다운로드 방식을 사용합니다. 브라우저 자동화로 엑셀 파일을 다운로드하고, 판다스(Pandas)로 파일을 파싱하고 구조화합니다.

백업으로는 직접 테이블 수집 방식을 사용합니다. 셀레니움(Selenium) 헤드리스 브라우저(Headless Browser)를 사용해서 자바스크립트가 완전히 로딩될 때까지 대기한 후 DOM 돔(Document Object Model)을 파싱합니다. 테이블 구조를 보존하면서 JSON 제이슨으로 변환합니다.

안정성 확보를 위해 모듈화된 크롤러 구조로 기능별로 분리했고, 에러 핸들링과 자동 재시도 기능을 넣었으며, 레거시 크롤러를 백업 시스템으로 유지하고 있습니다.

## 6. 핵심 기술 - AI 분석 엔진 (Ollama)

두 번째 핵심 기술은 AI 에이아이 분석 엔진입니다.

### Ollama 로컬 LLM 활용

올라마(Ollama)라는 오픈소스 로컬 AI 에이아이를 사용해서 데이터 프라이버시를 보장합니다.

### 기술 스펙 및 제약사항

모델은 라마(Llama) 3.1의 8B 에잇비, 80억 파라미터(8B) 인스트럭트(Instruct) 버전을 사용하고, 로컬 서버에서 CPU 씨피유 전용으로 실행됩니다. GPU 지피유가 없습니다.

장점은 API 에이피아이 호출 비용이 전혀 들지 않고, 데이터가 외부로 유출되지 않는다는 것입니다.

제약사항으로는 16K 식스틴케이, 만육천 토큰 제한이 있고, CPU 씨피유 성능 제한이 있으며, 클로드(Claude)나 GPT 지피티 대비 분석 품질 차이가 있습니다.

극복 방법으로는 통계표별로 분할해서 분석해서 토큰 제한을 회피하고, CPU 씨피유에 최적화된 모델을 사용합니다.

### 분석 프로세스

6단계로 진행됩니다. 데이터 구조를 분석해서 컬럼, 연도, 단위를 파악하고, 기본 통계량을 계산합니다. 시계열 트렌드를 분석해서 증가나 감소 패턴을 파악하고, 주요 변화 시점을 탐지합니다. 통계표별로 개별 인사이트를 생성하고, 마지막으로 전체 통계의 의미를 해석하는 종합 인사이트를 만듭니다.

### 결과 및 현실

전문가가 아니어도 쉽게 이해할 수 있는 텍스트 인사이트를 제공하고, 데이터가 외부로 유출되지 않아 안전합니다. 구조화된 데이터로 추후 시각화도 가능합니다.

다만 성능 한계가 있습니다. 통계표당 3분에서 10분 정도 소요되고, 대형 통계는 더 오래 걸립니다. 실시간 분석보다는 배치 처리에 적합합니다.

품질 비교를 해보면 체감상 클로드(Claude)나 GPT 지피티 대비 30%에서 40% 정도 품질 차이가 있습니다. 기본 통계는 잘 뽑지만 복잡한 해석은 약합니다. 하지만 업무 목적인 통계 선정용으로는 충분합니다.

모니터링은 SSE 에스에스이로 실시간 진행률을 표시하고, 패스트API(FastAPI) 로그와 브라우저 콘솔로 확인합니다. 별도 모니터링 도구는 없습니다. 개인 프로젝트니까요.

세 번째 핵심 기술은 캐시 시스템입니다.

### 다층 캐시 구조

백엔드의 캐시 폴더 아래에 4개의 하위 폴더가 있습니다. 메타데이터(metadata) 폴더에는 통계 메타데이터가, collected_data 콜렉티드데이터 폴더에는 수집된 원본 데이터가 JSON 제이슨으로 저장됩니다. analysis_results 어날리시스리절츠 폴더에는 AI 에이아이 분석 결과가, recent_stats 리센트스탯츠 폴더에는 최근 통계 목록이, tables 테이블즈 폴더에는 테이블 구조 데이터가 저장됩니다.

### 캐시 전략

응답 속도가 초 단위에서 밀리초 단위로 빨라지고, 불필요한 크롤링을 방지하며, 서버 부하가 감소합니다.

## 7. 프로젝트 구조

프로젝트 구조를 보시면, 크게 백엔드와 프론트엔드로 나뉩니다.

### 백엔드

백엔드는 패스트API(FastAPI) 서버로, app 폴더 아래에 api 에이피아이 폴더에 5개 모듈이 있습니다. 주요 분석 로직인 analysis_router 어날리시스라우터, 새로 추가된 AI 에이아이 채팅 API 에이피아이인 chat_router 챗라우터 등이 있습니다.

services 서비스즈 폴더에는 12개 서비스가 있는데, 크롤링 총괄인 crawler_service 크롤러서비스, AI 에이아이 인사이트와 채팅을 담당하는 ollama_service 올라마서비스, 새로 추가된 크로마DB 크로마디비 RAG 래그를 위한 vector_db_service 벡터디비서비스 등이 있습니다.

crawlers 크롤러즈 폴더에는 실제 사용 중인 modular_molit_crawler 모듈러몰릿크롤러가 있고, 레거시 폴더는 백업용입니다.

cache 캐시 폴더에는 캐시 저장소가 있고, data 데이터 폴더 아래 vector_db 벡터디비 폴더에는 새로 추가된 크로마DB 크로마디비 저장소가 있습니다.

### 프론트엔드

프론트엔드는 리액트(React) 앱으로, components 컴포넌츠 폴더에 10개 UI 유아이 컴포넌트가 있습니다. SSE 에스에스이를 담당하는 RealTimeProgressViewer 리얼타임프로그레스뷰어, 채팅 UI 유아이가 포함된 EnhancedBasicStatisticsViewer 인핸스드베이직스태티스틱스뷰어 등이 있습니다.

services 서비스즈 폴더의 api.ts 에이피아이닷티에스는 API 에이피아이 호출과 SSE 에스에스이를 담당합니다.

### MCP 서버

mcp-servers 엠씨피서버즈 폴더는 향후 확장용으로 현재는 미사용입니다.

별표 표시는 실제 사용 중인 핵심 파일이고, 별 두 개 NEW 뉴 표시는 채팅과 RAG 래그 기능 관련 파일입니다.

## 8. 개발 방법론 - 바이브 코딩

### 개발 규모

개발 기간은 2주에서 3주, 실제 작업 시간은 40시간에서 60시간 정도 걸렸습니다. 개발 인원은 1명이고, 디자이너 배경입니다. 주말과 퇴근 후 하루 2시간에서 3시간씩 작업했고, 산출물로 일자별 개발 일지 20여 개를 docs 폴더에 남겼습니다.

### 바이브 코딩이란?

바이브 코딩은 AI 에이아이와 대화하며 코드를 작성하는 방식입니다. 클로드(Claude), 커서 AI 에이아이(Cursor AI)와 실시간으로 대화하면서, 문제를 설명하면 AI 에이아이가 코드를 제안하고, 테스트하면서 개선해나갑니다. 전통적인 검색하고 복붙하는 방식이 아니라 협업 개발이라고 볼 수 있습니다.

### 바이브 코딩의 장점

비개발자도 복잡한 시스템을 구축할 수 있습니다. 디자이너인 제가 풀스택 개발을 완성했고, 전통적 방식 대비 2배에서 3배 빠릅니다.

프론트엔드 개발 프로세스를 보면, 와이어프레임(Wireframe)을 손으로 그려서 클로드에게 설명하면, 리액트(React) 컴포넌트를 제안하고, 테스트하고 개선하는 방식입니다. 테일윈드 씨에스에스(Tailwind CSS)로 스타일링도 해줍니다.

디버깅 전략으로는 에러 메시지를 클로드에게 복붙하고, console.log 콘솔로그, print 프린트 문을 많이 쓰고, 크롬 개발자도구(Chrome DevTools), 패스트API(FastAPI)의 /docs 독스 페이지를 활용합니다. 문제를 작게 나누어 해결하는 것이 중요합니다.

### 한계점

AI 에이아이가 제안한 코드를 이해 없이 사용하면 위험합니다. 복잡한 버그는 직접 디버깅이 필요합니다. 특히 비동기(Async), SSE 에스에스이, CORS 코르스(Cross-Origin Resource Sharing) 같은 문제는 직접 해결해야 합니다. 프로젝트 전체 구조 파악은 개발자 몫이고, 주석과 문서가 부족해서 향후 개선이 필요합니다.

## 9. 개발 과정 - 주요 챌린지

개발 과정에서 여러 챌린지가 있었습니다.

### 1. IBSheet 동적 테이블 크롤링

첫 번째 챌린지는 IBSheet 아이비시트 동적 테이블 크롤링이었습니다.

문제점은 특수 자바스크립트(JavaScript) 라이브러리로 렌더링되어서 DOM 돔에 직접 접근할 수가 없었습니다.

해결책은 셀레니움(Selenium)으로 완전히 로딩될 때까지 대기하고, 자바스크립트 실행이 완료된 후 DOM 돔을 파싱하는 것이었습니다.

### 2. 대용량 데이터 처리

두 번째 챌린지는 대용량 데이터 처리였습니다.

수십 년치 통계 데이터를 처리하다 보니 메모리 부족 위험이 있었습니다.

해결책은 캐시 시스템을 구축하고, 점진적 로딩, 레이지 로딩(Lazy Loading)을 적용하고, 판다스(Pandas)를 최적화하는 것이었습니다.

## 9. 개발 과정 - 주요 챌린지 (계속)

### 3. 실시간 진행률 표시

세 번째 챌린지는 실시간 진행률 표시였습니다.

문제점은 작업 시간이 1분에서 2분 정도로 길어서 사용자가 이탈할 가능성이 있었습니다.

해결책은 SSE 에스에스이, 서버 센트 이벤츠(Server-Sent Events)를 도입해서 단계별 진행률을 실시간으로 전송하고, 취소 기능도 제공하는 것이었습니다.

### 4. AI 분석 - 하드웨어 제약 극복

네 번째 챌린지는 AI 에이아이 분석에서 하드웨어 제약을 극복하는 것이었습니다.

하드웨어 환경이 CPU 씨피유만 있는 노트북이라 GPU 지피유가 없고, 클로드(Claude)나 GPT 지피티 수준의 분석이 어려웠으며, 메모리 및 토큰 제한이 있었습니다.

문제점은 처음에 통계 전체 데이터를 한 번에 분석하려고 했는데, 토큰 양이 초과해서 실패했다는 것입니다. 16K 식스틴케이 토큰 제한 때문이었죠. 클로드나 GPT 지피티 API 에이피아이를 사용하는 것도 고려했지만 비용 부담이 있었습니다.

해결책은 올라마(Ollama) 로컬 엘엘엠(LLM)을 활용해서 비용을 0원으로 만들고, 핵심 해결책으로 통계표별 분할 분석을 도입한 것입니다. 통계 전체를 개별 통계표로 분할하고, 각 표별로 인사이트를 생성한 후 종합하는 방식입니다. 프롬프트(Prompt)도 최적화하고, CPU 씨피유 환경에 최적화된 모델을 선택했습니다.

## 10. 향후 개선 방향

### 단기 목표 (3개월)

단기 목표로는 세 가지가 있습니다.

첫째, 성능 개선이 최우선입니다. GPU 지피유 활용으로 CUDA 쿠다를 지원하고, 경량화 모델을 테스트하며, 캐싱 전략을 강화하고, 배치 처리를 최적화할 계획입니다.

둘째, AI 에이아이 채팅 기능을 고도화하겠습니다. 대화 맥락 이해를 개선하고, 질문 의도 파악을 강화하며, 차트나 표를 생성하는 답변도 지원할 예정입니다.

셋째, 데이터 시각화를 구현하겠습니다. HTML 에이치티엠엘 기반 차트 생성으로 Chart.js 차트제이에스나 D3.js 디쓰리제이에스를 사용하고, 카드뉴스 형태 레이아웃을 설계하며, 인사이트와 차트를 통합한 뷰를 만들 계획입니다.

더 많은 통계 카테고리를 지원하고, 통계 간 비교 기능도 추가할 예정입니다.

### 중기 목표 (6개월)

중기 목표로는 카드뉴스를 완전히 구현하는 것이 최우선입니다. 시각화된 분석 결과를 제공하고, PDF 피디에프나 이미지로 내보내는 기능, 소셜 미디어 공유 기능을 만들 계획입니다.

타 정부기관 통계로 확장해서 통계청, 산업부 등으로 범위를 넓히고, 사용자 맞춤형 대시보드와 통계 알림 서비스도 만들 예정입니다. 새 통계가 발표되면 알림을 받는 거죠.

### 장기 목표 (1년)

장기 목표로는 모바일 앱을 개발하고, 리액트 네이티브(React Native)를 사용할 계획입니다. 다국어 지원으로 영어, 중국어를 추가하고, 개발자용 오픈 API 에이피아이도 제공할 예정입니다.

## 11. 프로젝트 성과

### 기술적 성과

기술적으로는 많은 성과가 있었습니다.

완전 자동화된 크롤링 시스템을 구축했고, 실시간 진행률 표시를 SSE 에스에스이로 구현했습니다. 로컬 AI 에이아이 기반 분석 엔진을 통합하고, 안정적인 캐시 시스템을 구축했습니다.

새롭게 RAG 래그 기반 질의응답 시스템을 크로마DB 크로마디비와 올라마(Ollama)로 구축했고, 벡터 검색 기반으로 정확도를 향상시켰습니다. 데이터 구조화 및 저장 시스템을 완성했고, 토큰 제한 문제를 통계표별 분할 전략으로 해결했습니다.

### 사용자 경험 개선

사용자 경험도 크게 개선되었습니다.

통계당 5분에서 10분이면 자동으로 수집되고 인사이트가 생성됩니다. 직관적인 UI 유아이, UX 유엑스를 제공하고, 실시간 진행률 피드백으로 투명성을 확보했습니다. GPU 지피유 없이 CPU 씨피유만으로 동작합니다.

### 개발 경험

개발 경험도 흥미로웠습니다.

비개발자인 디자이너가 풀스택 개발을 완성했고, 바이브 코딩으로 2주에서 3주 만에, 실제 작업 시간 40시간에서 60시간 만에 구현했습니다. 코드 품질도 모듈화된 구조, 타입 안정성, 에러 핸들링을 갖추고 있고, 문서화도 일자별 개발 일지 20여 개를 작성했습니다.

### 타 팀 사용 시 고려사항

다만 타 팀에서 사용하려면 몇 가지 개선이 필요합니다.

현재 장벽으로는 셋업(Setup) 자동화가 안 되어 있고, 주석과 문서가 부족하며, 파이썬(Python), 타입스크립트(TypeScript) 기본 지식이 필요합니다.

개선이 필요한 부분은 도커 컴포즈(Docker Compose) 자동화, 사용자 매뉴얼 작성, 에러 메시지를 친절하게 만드는 것입니다.

### 현재 단계

현재는 1단계가 완료되어서 데이터 수집과 AI 에이아이 인사이트까지 되고, 2단계인 시각화와 카드뉴스는 진행 예정입니다.

## 12. 핵심 가치 제안

### 1. 실무자 관점의 실용성

첫 번째 핵심 가치는 실무자 관점의 실용성입니다.

"통계이야기" 제작자를 위한 실용 도구로서, 통계 분석 전문 지식이 없어도 AI 에이아이가 자동으로 핵심 인사이트를 추출해주고, 어떤 통계를 선택할지 의사결정을 지원합니다.

### 2. 시간 배분 최적화

두 번째는 시간 배분 최적화입니다.

기존에는 통계를 하나하나 수동으로 분석해야 했지만, 개선 후에는 밤사이 자동으로 분석하고 아침에 결과를 확인하면 됩니다. 실제 소요 시간은 통계당 5분에서 10분이지만, 사람이 직접 작업할 필요는 없습니다.

### 3. 기획 효율성 향상

세 번째는 기획 효율성 향상입니다.

AI 에이아이 인사이트로 기획 방향을 빠르게 수립할 수 있고, 통계별 핵심 포인트를 한눈에 비교할 수 있으며, 카드뉴스 소재 선정 시간이 단축됩니다.

### 4. 확장 가능성

네 번째는 확장 가능성입니다.

국토교통부에서 시작해서 모든 정부 통계로 확장할 수 있고, 인사이트에서 시각화 카드뉴스로 발전할 수 있으며, 개인 프로젝트에서 실무 도구로 진화할 수 있습니다.

## 13. 현재 한계점 & 개선 계획

### 주요 한계점

현재 한계점도 솔직하게 말씀드리겠습니다.

첫째, 성능 문제가 가장 큰 이슈입니다. 통계표당 3분에서 10분이 소요되고, CPU 씨피유 환경이라 대형 통계는 10개 이상 표가 있으면 1시간 이상 걸립니다. 실시간 사용이 불가능합니다.

둘째, 분석 품질이 클로드(Claude)나 GPT 지피티 대비 차이가 있습니다. 복잡한 통계 해석에는 한계가 있고, 한국어 이해도도 제한적입니다.

셋째, 사용성 측면에서 밤사이 배치 처리가 필요하고, 즉각적인 피드백이 어렵습니다.

### 개선 계획

개선 계획으로는 세 가지 단계가 있습니다.

단기로 1개월에서 2개월 안에 GPU 지피유 서버를 확보하거나 클라우드 GPU 지피유를 활용하고, 모델 양자화 및 최적화, 병렬 처리를 개선할 계획입니다.

중기로 3개월에서 6개월 안에 더 효율적인 경량 모델을 테스트하고, 중요한 통계만 클로드나 GPT 지피티 API 에이피아이를 하이브리드로 사용하며, 사전 학습 데이터를 구축할 예정입니다.

장기적으로는 전용 서버를 구축해서 실시간 분석이 가능한 수준으로 개선하겠습니다.

### 그럼에도 가치가 있는 이유

그럼에도 가치가 있는 이유는, 완전 자동화라서 사람 개입 없이 분석이 되고, API 에이피아이 비용이 0원으로 비용 절감이 되며, 배치 처리로 밤사이 돌려놓으면 되고, MVP 엠비피, 미니멈 바이어블 프로덕트(Minimum Viable Product)로 컨셉이 작동한다는 것을 검증했기 때문입니다.

## 14. 배운 점 & 인사이트

### 기술적 학습

기술적으로 많은 것을 배웠습니다.

바이브 코딩으로 AI 에이아이, 클로드와 대화하며 개발하는 방법을 익혔고, 동적 웹 크롤링 노하우를 습득했습니다. 패스트API(FastAPI)와 리액트(React)로 풀스택 개발 경험을 쌓았고, SSE 에스에스이 실시간 통신을 구현했습니다.

로컬 엘엘엠(LLM) 통합과 제약사항 극복 방법을 배웠고, 올라마(Ollama)를 사용했습니다. 엑셀 자동화 및 파싱 기술도 익혔고, 토큰 제한 문제를 데이터 분할 전략으로 해결하는 법을 배웠습니다.

새롭게 RAG 래그, 리트리벌 어그먼티드 제너레이션(Retrieval-Augmented Generation) 구현 경험을 쌓았고, 벡터 DB 디비, 크로마DB 크로마디비(ChromaDB) 통합 및 의미 검색 기술도 익혔습니다.

### 프로젝트 관리

프로젝트 관리 측면에서는 점진적 개선의 중요성을 배웠습니다. MVP 엠비피에서 시작해서 기능을 추가하는 방식이죠. 사용자 피드백을 반영하는 것과 문서화, 개발 과정을 기록하는 것의 중요성도 깨달았습니다.

### 비즈니스 인사이트

비즈니스 인사이트로는 여러 가지가 있습니다.

실무 문제에서 출발한 프로젝트의 가치를 느꼈고, 공공 데이터의 활용 가능성이 무한하다는 것을 알았습니다. AI 에이아이는 도구일 뿐이고 문제 정의가 핵심이라는 것, 비개발자도 필요에 의해 개발이 가능하다는 것, 바이브 코딩 덕분이죠.

"내가 불편한 것은 다른 사람도 불편하다"는 것을 배웠고, 제약사항도 창의적으로 해결 가능하다는 것, 토큰 제한을 분할 전략으로 해결한 것처럼요.

완벽하지 않아도 MVP 엠비피로 가치 검증이 가능하고, 한계를 솔직히 인정하는 것이 더 신뢰를 준다는 것도 배웠습니다.

## 15. Q&A

자주 묻는 질문 6가지를 준비했습니다.

### Q1. 왜 이 프로젝트를 만들게 되었나요?

매달 "통계이야기" 카드뉴스를 제작해야 하는데, 한 달치 통계를 일일이 확인하고 분석하는 게 너무 불편했습니다. 디자이너 배경이라 통계 분석 경험도 없어서, AI 에이아이가 인사이트만 뽑아줘도 기획 방향을 잡기 훨씬 수월할 것 같아 시작했습니다.

### Q2. 실제로 업무에 활용하고 있나요?

현재 1단계인 데이터 수집과 인사이트까지 완료했습니다. 다만 통계당 5분에서 10분 걸려서 실시간 활용은 어렵고, 전날 밤 배치로 돌려놓고 아침에 결과를 확인하는 방식으로 활용 중입니다. 성능 개선이 필요한 상태입니다.

### Q3. 크롤링이 불법이 아닌가요?

공공데이터는 개방 원칙에 따라 활용 가능합니다. 서버에 부담을 주지 않도록 적절한 대기 시간을 두고 있습니다.

### Q4. AI 분석 비용은 얼마나 드나요?

올라마(Ollama) 로컬 엘엘엠(LLM)을 사용하여 API 에이피아이 비용이 전혀 들지 않습니다. GPU 지피유 없이 CPU 씨피유만으로도 동작하지만, 통계표당 3분에서 10분 소요되어 성능이 느린 것이 단점입니다. CPU 씨피유 노트북에서도 동작하는 완전 로컬 솔루션으로 만들었고, 품질 차이는 있지만 업무에는 충분히 활용 가능한 수준입니다.

### Q5. 개발 경험이 없는데 어떻게 만들었나요?

바이브 코딩으로 개발했습니다. 클로드 AI 에이아이, 커서(Claude AI, Cursor)와 대화하며 "이런 기능을 만들고 싶다"고 설명하면 AI 에이아이가 코드를 제안하고, 테스트하면서 개선해나갔습니다. 물론 완전 자동은 아니고 AI 에이아이 제안을 이해하고 디버깅하는 과정이 필요했습니다.

### Q6. AI 채팅 기능은 어떻게 작동하나요?

RAG 래그, 리트리벌 어그먼티드 제너레이션(Retrieval-Augmented Generation) 방식입니다. 질문이 들어오면 크로마DB 크로마디비(ChromaDB)에서 벡터 유사도로 관련 데이터만 상위 8개를 검색하고, AI 에이아이 인사이트와 함께 올라마(Ollama)에게 전달합니다. 전체 데이터가 아닌 필요한 부분만 사용해서 토큰 제한도 회피하고 정확도도 높아집니다.
